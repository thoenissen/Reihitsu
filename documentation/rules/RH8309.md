# RH8309 — XML documentation elements must follow a prescribed order

| Property | Value |
|----------|-------|
| **ID** | RH8309 |
| **Category** | Documentation |
| **Severity** | Warning |
| **Code Fix** | ✓ |

## Description

The top-level elements of an XML documentation comment must appear in a fixed, conventional order:

1. `<summary>`
2. `<typeparam>` (in declaration order)
3. `<param>` (in declaration order)
4. `<returns>`
5. `<value>`
6. `<exception>`
7. `<permission>`
8. `<example>`
9. `<remarks>`

Unknown or custom elements (for example `<inheritdoc>`, `<seealso>` or `<include>`) are tolerated and do not trigger the rule. The code fix keeps such elements in their original positions and only reorders the canonical elements around them.

## Why is this a problem?

When documentation elements always follow the same order, documentation blocks become easier to scan, compare, and review. Mixed ordering — for example a `<returns>` before the `<summary>`, or a `<param>` before the `<summary>` — forces the reader to hunt for the part they need and produces noisier diffs.

## How to fix it

Reorder the documentation elements so they follow the canonical order shown above. The code fix performs the reordering automatically while preserving each element's content.

## Examples

### Violation

```cs
internal class TestClass
{
    /// <returns>The sum.</returns>
    /// <summary>
    /// Adds two numbers.
    /// </summary>
    /// <param name="a">First value.</param>
    /// <param name="b">Second value.</param>
    public int Add(int a, int b)
    {
        return a + b;
    }
}
```

### Correction

```cs
internal class TestClass
{
    /// <summary>
    /// Adds two numbers.
    /// </summary>
    /// <param name="a">First value.</param>
    /// <param name="b">Second value.</param>
    /// <returns>The sum.</returns>
    public int Add(int a, int b)
    {
        return a + b;
    }
}
```
