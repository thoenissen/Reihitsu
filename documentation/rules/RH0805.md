# RH0805 — Override indexers should be grouped by base type regions

| Property | Value |
|----------|-------|
| **ID** | RH0805 |
| **Category** | Formatting |
| **Severity** | Warning |
| **Code Fix** | ❌ |

## Description

This rule requires each `override` indexer to appear inside a top-level `#region` whose label matches the type that originally introduced the overridden member.

## Why is this a problem?

Override indexers are easier to scan when they are grouped by the type that originally introduced them. Generic region names such as `Indexers` hide that relationship and make inheritance-heavy types harder to navigate.

## How to fix it

Move each `override` indexer into a top-level region named after the type where the overridden member was first declared. Use the type name only, such as `BaseProcessor`.

## Examples

### Violation

```cs
internal abstract class BaseProcessor
{
    public virtual string this[int index]
    {
        get
        {
            return string.Empty;
        }
        set
        {
        }
    }
}

internal class DerivedProcessor : BaseProcessor
{
    #region Indexers

    public override string this[int index]
    {
        get
        {
            return base[index];
        }
        set
        {
            base[index] = value;
        }
    }

    #endregion // Indexers
}
```

### Correction

```cs
internal abstract class BaseProcessor
{
    public virtual string this[int index]
    {
        get
        {
            return string.Empty;
        }
        set
        {
        }
    }
}

internal class DerivedProcessor : BaseProcessor
{
    #region BaseProcessor

    public override string this[int index]
    {
        get
        {
            return base[index];
        }
        set
        {
            base[index] = value;
        }
    }

    #endregion // BaseProcessor
}
```
