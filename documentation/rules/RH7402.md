# RH7402 — Override properties should be grouped by base type regions

| Property | Value |
|----------|-------|
| **ID** | RH7402 |
| **Category** | Organization |
| **Severity** | Warning |
| **Code Fix** | ❌ |

## Description

This rule requires each `override` property to appear inside a top-level `#region` whose label matches the type that originally introduced the overridden member.

An override that also implements an interface property remains governed by this rule. RH7410 does not apply to it, so the property belongs to the base-type region rather than the interface region.

## Why is this a problem?

Override properties are easier to scan when they are grouped by the type that originally introduced them. Generic region names such as `Properties` hide that relationship and make inheritance-heavy types harder to navigate.

## How to fix it

Move each `override` property into a top-level region named after the type where the overridden member was first declared. Use the type name only, such as `BaseProcessor`.

## Examples

### Violation

```cs
internal abstract class BaseProcessor
{
    public virtual string Name => string.Empty;
}

internal class DerivedProcessor : BaseProcessor
{
    #region Properties

    public override string Name => base.Name;

    #endregion // Properties
}
```

### Correction

```cs
internal abstract class BaseProcessor
{
    public virtual string Name => string.Empty;
}

internal class DerivedProcessor : BaseProcessor
{
    #region BaseProcessor

    public override string Name => base.Name;

    #endregion // BaseProcessor
}
```
