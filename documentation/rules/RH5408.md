# RH5408 — Auto-properties should be single lined

| Property     | Value        |
|--------------|--------------|
| **ID**       | RH5408       |
| **Category** | Layout |
| **Severity** | Warning      |
| **Code Fix** | ✓ |

## Description

This rule checks that brace-based auto-properties whose accessors have no body are written on a single line whenever the declaration can be simplified safely.

The rule applies to auto-properties such as `get;`, `get; set;`, and `get; init;`. Attributes do not exclude a property from the rule. Property-level and accessor-level attributes may still be simplified to a single-line declaration.

## Why is this a problem?

A multi-line auto-property adds vertical noise without adding meaning. Simple accessor declarations are easier to scan when the full property stays compact, especially when a type contains many small members.

A declaration like

```cs
[Example]
public int Count
{
    [Example]
    get;
    [Example]
    set;
}
```

is harder to read than the equivalent single-line form because the accessor list carries no implementation logic.

## How to fix it

Collapse the declaration so the property signature and accessor list are on one line. Keep property attributes on their own line when they already precede the declaration, but collapse the accessor list itself.

Auto-properties are not reported when a comment or directive would be crossed by the collapse — whether it sits inside the accessor list or in the gap between the signature and the accessor brace (for example `public int Value // note` on its own line). The formatter refuses to join across that trivia, so the code fix only applies when collapsing can be done safely.

## Examples

### Violation

```cs
[Example]
public int Count
{
    [Example]
    get;
    [Example]
    set;
}

public string Name
{
    get;
}
```

### Correction

```cs
[Example]
public int Count { [Example] get; [Example] set; }

public string Name { get; }
```
