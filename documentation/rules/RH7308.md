# RH7308 — Standard regions should contain only their matching member kind

| Property | Value |
|----------|-------|
| **ID** | RH7308 |
| **Category** | Organization |
| **Severity** | Warning |
| **Code Fix** | ❌ |

## Description

This rule enforces that a top-level region using one of the conventional labels contains only members of the kind implied by that label. The recognized member-kind nouns and the declarations they imply are:

| Region noun | Expected member kind |
|-------------|----------------------|
| `Fields`, `Constants` | field declarations |
| `Properties` | property declarations |
| `Methods` | method declarations |
| `Constructors` | constructor declarations |
| `Finalizer`, `Destructor` | finalizer declarations |
| `Events` | event declarations |
| `Indexers` | indexer declarations |
| `Operators` | operator declarations |

Label matching is flexible and case-insensitive:

- Both singular and plural nouns are recognized (`Field` and `Fields`).
- Access and other qualifiers are ignored, so `Private methods`, `Public methods`, or `Static fields` are all recognized by their noun.
- Combined labels list several allowed kinds, separated by `/`, `&`, `,`, or the words `and`/`or`. For example, members in a `Constructors / Finalizer` region may be constructors or finalizers.

Only members placed directly inside the region are considered. Nested types (including delegates and enums) are out of scope, and labels that contain no recognized member-kind noun (for example, a region named after an interface) are ignored.

## Why is this a problem?

When a type uses the conventional region names, readers expect each region to hold only members of that kind. A `Fields` region that also contains a property or a method defeats the purpose of the region and makes the type harder to scan.

## How to fix it

Move the flagged member into the region that matches its kind, creating that region if it does not exist yet. For example, move a property out of a `Fields` region and into a `Properties` region.

## Examples

### Violation

```cs
internal class TestClass
{
    #region Fields

    private int _value;

    public int Value => _value;

    #endregion // Fields
}
```

### Correction

```cs
internal class TestClass
{
    #region Fields

    private int _value;

    #endregion // Fields

    #region Properties

    public int Value => _value;

    #endregion // Properties
}
```
