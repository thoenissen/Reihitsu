# RH7308 — Standard regions should contain only their matching member kind

| Property | Value |
|----------|-------|
| **ID** | RH7308 |
| **Category** | Organization |
| **Severity** | Warning |
| **Code Fix** | ❌ |

## Description

This rule enforces that a top-level region using one of the conventional labels contains only members of the kind implied by that label. The recognized labels and their member kinds are:

| Region label | Expected member kind |
|--------------|----------------------|
| `Fields` | field declarations |
| `Properties` | property declarations |
| `Methods` | method declarations |
| `Events` | event declarations |
| `Constructors` | constructor declarations |

Only members placed directly inside the region are considered. Nested types are out of scope.

## Why is this a problem?

When a type uses the conventional region names, readers expect each region to hold only members of that kind. A `Fields` region that also contains a property or a method defeats the purpose of the region and makes the type harder to scan.

## How to fix it

Move the flagged member into the region that matches its kind, creating that region if it does not exist yet. For example, move a property out of a `Fields` region and into a `Properties` region.

## Examples

### Violation

```cs
internal class TestClass
{
    #region Fields

    private int _value;

    public int Value => _value;

    #endregion // Fields
}
```

### Correction

```cs
internal class TestClass
{
    #region Fields

    private int _value;

    #endregion // Fields

    #region Properties

    public int Value => _value;

    #endregion // Properties
}
```
