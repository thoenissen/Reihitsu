# RH7311 — Region descriptions must be unique within a type

| Property     | Value        |
|--------------|--------------|
| **ID**       | RH7311       |
| **Category** | Organization |
| **Severity** | Warning      |
| **Code Fix** | ❌            |

## Description

This rule checks that no two `#region` directives inside the same type declaration share the same description. The comparison is case-insensitive and ignores surrounding whitespace. Identical descriptions in nested types or in other files are not affected — only sibling regions of one type are compared. The diagnostic is reported on the second (and any further) region that repeats a description, so the first occurrence stays as the canonical block.

## Why is this a problem?

Duplicate region names defeat the purpose of regions. The collapse tree and the navigation drop-down become ambiguous, because two entries carry the same label. A repeated description also usually signals that members which belong together were split across two blocks. A distinct name per region keeps navigation unambiguous and encourages related members to live in a single block.

## How to fix it

Either merge the two blocks into a single region when they group the same kind of members, or give one of them a distinct, meaningful description. There is no automated code fix, because merging and renaming both depend on intent that cannot be inferred safely.

## Examples

### Violation

```cs
public class C
{
    #region Methods

    public void First() { }

    #endregion // Methods

    #region Methods

    public void Second() { }

    #endregion // Methods
}
```

### Correction

```cs
public class C
{
    #region Methods

    public void First() { }

    public void Second() { }

    #endregion // Methods
}
```
