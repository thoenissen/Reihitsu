# RH0414 - Enum members in non-private enums must be documented

| Property | Value |
|----------|-------|
| **ID** | RH0414 |
| **Category** | Documentation |
| **Severity** | Warning |
| **Code Fix** | ❌ |

## Description

This rule requires enum members in non-private enums to have XML documentation. A member counts as documented when it has a `<summary>` tag or uses `<inheritdoc/>`.

## Why is this a problem?

Undocumented enum members in non-private enums make it harder to understand what each value represents. Readers have to infer intent from the enum name alone, which is often not enough.

## How to fix it

Add a meaningful XML comment above each enum member in the non-private enum. In most cases a short `<summary>` is the best fix.

## Examples

### Violation

```cs
internal enum OrderState
{
    Open,
}
```

### Correction

```cs
internal enum OrderState
{
    /// <summary>
    /// The order was created and is waiting for processing
    /// </summary>
    Open,
}
```
