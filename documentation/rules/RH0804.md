# RH0804 — Override events should be grouped by base type regions

| Property | Value |
|----------|-------|
| **ID** | RH0804 |
| **Category** | Formatting |
| **Severity** | Warning |
| **Code Fix** | ❌ |

## Description

This rule requires each `override` event to appear inside a top-level `#region` whose label matches the type that originally introduced the overridden member.

## Why is this a problem?

Override events are easier to scan when they are grouped by the type that originally introduced them. Generic region names such as `Events` hide that relationship and make inheritance-heavy types harder to navigate.

## How to fix it

Move each `override` event into a top-level region named after the type where the overridden member was first declared. Use the type name only, such as `BaseProcessor`.

## Examples

### Violation

```cs
internal abstract class BaseProcessor
{
    public abstract event System.EventHandler Completed;
}

internal class DerivedProcessor : BaseProcessor
{
    #region Events

    public override event System.EventHandler Completed;

    #endregion // Events
}
```

### Correction

```cs
internal abstract class BaseProcessor
{
    public abstract event System.EventHandler Completed;
}

internal class DerivedProcessor : BaseProcessor
{
    #region BaseProcessor

    public override event System.EventHandler Completed;

    #endregion // BaseProcessor
}
```
