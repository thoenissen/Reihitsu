# RH5113 — Declaration semicolon must stay on the declaration line

| Property | Value |
|----------|-------|
| **ID** | RH5113 |
| **Category** | Layout |
| **Severity** | Warning |
| **Code Fix** | ✔ |

## Description

This rule requires the terminating semicolon of a member declaration to appear on the same line as the end of the declaration. Field declarations, event field declarations, and delegate declarations are covered.

## Why is this a problem?

A semicolon broken onto its own line breaks the visual association between the declaration and its terminator. Editors and diff tools report a misleading second line that carries no information, and the declaration is harder to scan because the reader has to skip past empty space to confirm where it ends.

## How to fix it

Move the semicolon onto the line where the declaration ends, removing the intervening line break and any whitespace before the semicolon. The code fix delegates final layout to the formatter.

When a comment or directive sits in the gap between the declaration and its terminating semicolon, the shape is not reported, because the formatter refuses to collapse across that trivia.

## Examples

### Violation

```cs
internal class TestClass
{
    private int _value
        ;

    public event System.EventHandler Changed
        ;

    public delegate void Handler(int value)
        ;
}
```

### Correction

```cs
internal class TestClass
{
    private int _value;

    public event System.EventHandler Changed;

    public delegate void Handler(int value);
}
```
