# RH0809 — Non-override indexers should not be placed in override regions

| Property | Value |
|----------|-------|
| **ID** | RH0809 |
| **Category** | Formatting |
| **Severity** | Warning |
| **Code Fix** | ❌ |

## Description

This rule forbids non-override indexers from being placed inside a top-level override region whose label is used by override indexers in the same type.

## Why is this a problem?

Override regions are intended to group inherited members by their original base type. Placing ordinary indexers inside those regions obscures which members actually participate in the override surface of the type.

## How to fix it

Move each non-override indexer out of the override region and into a normal organizational region such as `Indexers`, or leave it ungrouped if that matches the surrounding style.

## Examples

### Violation

```cs
internal abstract class BaseProcessor
{
    public virtual string this[int index]
    {
        get
        {
            return string.Empty;
        }
    }
}

internal sealed class ReportProcessor : BaseProcessor
{
    #region BaseProcessor

    public override string this[int index]
    {
        get
        {
            return base[index];
        }
    }

    public string this[string key]
    {
        get
        {
            return key;
        }
    }

    #endregion // BaseProcessor
}
```

### Correction

```cs
internal abstract class BaseProcessor
{
    public virtual string this[int index]
    {
        get
        {
            return string.Empty;
        }
    }
}

internal sealed class ReportProcessor : BaseProcessor
{
    #region BaseProcessor

    public override string this[int index]
    {
        get
        {
            return base[index];
        }
    }

    #endregion // BaseProcessor

    #region Indexers

    public string this[string key]
    {
        get
        {
            return key;
        }
    }

    #endregion // Indexers
}
```
