## 1. General Principles

### 1.1 Indentation Unit

- **4 spaces** per indentation level.
- **No tabs.** All indentation is space-based.

### 1.2 Line Endings

- The formatter **preserves** the predominant line-ending style of the file (`\r\n` or `\n`).
- If no line endings are present, the system default (`Environment.NewLine`) is used.

### 1.3 Skipped Files

- Files with **syntax errors** are not formatted.
- Files with an **`<auto-generated />`** marker in a comment are not formatted.

### 1.4 Idempotency

- Running the formatter twice on any input **must** produce the same output as running it once.

## 2. Structural Transforms

Structural transforms change the syntactic shape of the code without altering its semantics. They run **first** in the pipeline, before any spacing or indentation rules.

### 2.1 Expression-Bodied Methods → Block Body

**Rule**: Methods **must not** use expression-body syntax (`=>`). They are converted to block-body form.

```csharp
// Before
public int GetValue() => 42;

// After
public int GetValue()
{
    return 42;
}
```

```csharp
// Before (void method)
public void DoWork() => Console.WriteLine("hello");

// After
public void DoWork()
{
    Console.WriteLine("hello");
}
```

**Details**:
- Void methods: expression becomes an `ExpressionStatement`.
- Non-void methods: expression is wrapped in a `ReturnStatement`.
- `async` methods, generic methods, static methods, and methods with parameters are all handled correctly.

### 2.2 Expression-Bodied Constructors → Block Body

**Rule**: Constructors **must not** use expression-body syntax. They are converted to block-body form.

```csharp
// Before
public MyClass(int x) => _x = x;

// After
public MyClass(int x)
{
    _x = x;
}
```

**Details**:
- Constructor initializers (`: this()`, `: base()`) are preserved.
- The expression always becomes an `ExpressionStatement` (constructors have no return value).

### 2.3 Expression-Bodied Get-Only Properties → Single Line

**Rule**: Get-only properties using expression-body syntax (`=>`) **are allowed** but **must be single-line**. Multi-line expression-bodied properties are collapsed to a single line.

```csharp
// Correct (stays as-is)
public string Name => _name;

// Before (multi-line)
public string Name
    => _name;

// After (collapsed)
public string Name => _name;
```

### 2.4 Expression-Bodied Operators, Indexers, Conversion Operators, Finalizers, Local Functions → Block Body

**Rule**: All remaining expression-bodied members are converted to block-body form, following the same rules as methods:

- **Operators**: `public static MyType operator +(MyType a, MyType b) => ...` → block body with `return`.
- **Indexers**: `public int this[int i] => ...` → block body with `return`.
- **Conversion operators**: `public static implicit operator int(MyType x) => ...` → block body with `return`.
- **Finalizers**: `~MyClass() => Cleanup();` → block body with expression statement.
- **Local functions**: `int Local() => 42;` → block body with `return`.

```csharp
// Before
public static MyType operator +(MyType a, MyType b) => new MyType(a.Value + b.Value);

// After
public static MyType operator +(MyType a, MyType b)
{
    return new MyType(a.Value + b.Value);
}
```

## 3. Brace Placement

### 3.1 Allman Style

**Rule**: Opening and closing braces are placed on **dedicated lines** (Allman style).

```csharp
// Correct (Allman)
public class MyClass
{
    public void Method()
    {
        if (condition)
        {
            DoWork();
        }
    }
}

// Incorrect (K&R)
public class MyClass {
    public void Method() {
    }
}
```

**Applies to**:
- Method/constructor/finalizer bodies (`BlockSyntax`)
- Type declarations (classes, structs, interfaces, records)
- Namespace declarations (non-file-scoped)
- `if`, `for`, `while`, `foreach`, `try`, `catch`, `finally`, `switch`, `lock`, `using`, `checked`, `unchecked`, `fixed` blocks
- Object initializers and collection initializers
- Anonymous object creation expressions

### 3.2 Auto-Property Exception

**Rule**: Auto-properties keep braces on the same line.

```csharp
// Correct — stays compact
public int Value { get; set; }
public string Name { get; }

// Not applicable — has body, uses Allman
public int Value
{
    get { return _value; }
    set { _value = value; }
}
```

### 3.3 Close-Brace Continuation

**Rule**: After a closing brace, a line break is **required** — except when the next token is `;`, `,`, or `)`.

```csharp
// Semicolon allowed on same line
var x = new Foo
        {
            A = 1
        };

// Comma allowed on same line
var items = new[]
            {
                new Foo { A = 1 },
                new Foo { A = 2 }
            };
```

### 3.4 First Member/Statement After Opening Brace

**Rule**: The first member or statement after `{` must be on a **new line**.

```csharp
// Incorrect
if (condition) { DoWork(); }

// Correct
if (condition)
{
    DoWork();
}
```

## 4. Indentation

### 4.1 Block Indentation

**Rule**: Each nesting construct increases indentation by **one level** (4 spaces).

**Indenting constructs**:
- Namespace declarations (non-file-scoped)
- Type declarations (class, struct, interface, record, enum)
- Block statements (`{ }`)
- Switch statement body
- Switch section (`case` label body)
- Accessor bodies (`get { }`, `set { }`)

**Non-indenting constructs**:
- File-scoped namespaces (`namespace X;`)
- `CompilationUnit` (the root)

```csharp
namespace MyApp                      // Level 0
{
    public class MyClass             // Level 1
    {
        public void Method()         // Level 2
        {
            if (condition)           // Level 3
            {
                DoWork();            // Level 4
            }
        }
    }
}
```

```csharp
// File-scoped namespace — no indent increase
namespace MyApp;

public class MyClass                 // Level 0
{
    public void Method()             // Level 1
    {
    }
}
```

### 4.2 Constructor Initializer Placement

**Rule**: Constructor initializers (`: base()`, `: this()`) are placed on a **new line**, indented by **+1 level** from the constructor.

```csharp
public MyClass(int x)
    : base(x)
{
}
```

### 4.3 Generic Constraint Placement

**Rule**: `where` constraint clauses are placed on **new lines**, each indented by **+1 level** from the declaration.

```csharp
public class MyClass<T>
    where T : class, IDisposable
{
}

public void Method<T, U>()
    where T : class
    where U : struct
{
}
```

## 5. Continuation-Line Alignment

Continuation-line alignment handles multi-line expressions where subsequent lines must be aligned to a specific column determined by the surrounding syntax — not just block indentation.

### 5.1 Argument/Parameter Alignment

**Rule**: When method arguments span multiple lines, the **first argument** stays on the same line as the opening parenthesis, and subsequent arguments align to the column after the opening parenthesis.

```csharp
var result = MyMethod(firstArg,
                      secondArg,
                      thirdArg);
```

**Rule**: If the first argument is on a new line, it is **collapsed** back to the same line as the opening parenthesis.

**Applies to**: method arguments, bracketed arguments (indexers), attribute arguments, method parameter declarations.

### 5.2 Method Chain Alignment

**Rule**: In multi-line method chains, dots (`.`, `?.`) align to the column of the first invocation dot.

```csharp
var result = items.Where(x => x.IsValid)
                  .Select(x => x.Name)
                  .OrderBy(x => x)
                  .ToList();
```

**Details**:
- The **first chain link** is collapsed to the same line as the root expression.
- Only **invoked** member accesses count as chain links (`.Foo()` — not `.Property`).
- Conditional access (`?.`) is treated as a chain link.
- Postfix operators (`!`) are preserved alongside dot tokens.
- Single-line chains are not reformatted.

**Note**: All chains are treated uniformly — no special case for single-link chains with statement-lambda arguments. The first link always collapses to the root line.

### 5.3 Object Initializer Alignment

**Rule**: Object initializer braces align to the column of the `new` keyword. Member assignments are indented by **+1 level** from the braces.

```csharp
var obj = new MyClass
          {
              Name = "Test",
              Value = 42
          };
```

**Details**:
- Nested initializers each create their own alignment context.
- Array initializers follow the same pattern.
- Collection initializers follow the same pattern.

### 5.4 Collection Expression Alignment (C# 12+)

**Rule**: Elements in collection expressions `[...]` are indented by **+1 level** from the opening bracket. The closing bracket aligns to the opening bracket column.

```csharp
var items = [
                new Item { Name = "A" },
                new Item { Name = "B" }
            ];
```

### 5.5 Binary Expression Alignment

**Rule**: When binary operators appear on continuation lines, they align to the column of the **left operand's first token**. The operator starts the continuation line, followed by the right operand.

```csharp
// Logical operators
if (conditionA
    && conditionB
    && conditionC)
{
}

var result = value1
             ?? value2
             ?? defaultValue;

// Arithmetic / bitwise operators
var total = basePrice
            + tax
            + shipping
            - discount;

var flags = optionA
            | optionB
            | optionC;
```

**Details**:
- Applies to **all** binary operators: logical (`&&`, `||`), null-coalescing (`??`), arithmetic (`+`, `-`, `*`, `/`, `%`), bitwise (`|`, `&`, `^`), shift (`<<`, `>>`), comparison (`==`, `!=`, `<`, `>`, `<=`, `>=`).
- Pattern operators `or` and `and` in `is` expressions follow the same rule.
- If an operator currently appears at the **end** of a line instead of the beginning of the next line, the formatter moves it to the beginning of the next line.

### 5.6 Conditional Expression (Ternary) Alignment

**Rule**: In multi-line conditional expressions, `?` and `:` are placed on new lines, aligned relative to the condition.

```csharp
var result = condition
                 ? trueValue
                 : falseValue;
```

**Nested ternaries**: Each nesting level adds +1 indent:

```csharp
return reader.Value != null
           ? reader.ValueType == typeof(long)
               ? TimeSpan.FromSeconds((long)reader.Value)
               : TimeSpan.FromSeconds((double)reader.Value)
           : TimeSpan.Zero;
```

### 5.7 Base Type List Alignment

**Rule**: In multi-line base type lists, types align to the column of the first base type.

```csharp
public class MyClass : IFoo,
                       IBar,
                       IBaz
{
}
```

### 5.8 Switch Expression Layout

**Rule**: Switch expression braces align to the governing expression column. Arms are indented by **+1 level**.

```csharp
var result = value switch
             {
                 1 => "one",
                 2 => "two",
                 _ => "other"
             };
```

**With `or` patterns**: Pattern combinators align to the arm's pattern column.

```csharp
var category = value switch
               {
                   1 or 2 or 3 => "low",
                   >= 4 and <= 7 => "medium",
                   > 7 or < 0 => "extreme",
                   _ => "other"
               };
```

**Details**:
- Pattern `or` and `and` operators in arms follow binary expression alignment rules (§5.5).
- Fluent chains in arms are handled separately from arm alignment.
- Object initializers in arms are rebuilt after arm positioning.

### 5.9 Lambda Formatting

**Rule**: Lambdas follow all standard formatting rules. There are no special cases for lambda parameters or bodies. The key rule is that when a lambda has a **block body**, the opening brace `{` is always placed on a new line aligned under the first argument (or `(` when there are multiple arguments, or `async` for async lambdas).

**Expression lambda in argument**:

```csharp
items.Where(x => x.Name != null
                  && x.Value > 0);
```

**Block lambda — single parameter (brace under first argument)**:

```csharp
items.ForEach(x =>
              {
                  Process(x);
                  Log(x);
              });
```

**Block lambda — multiple parameters (brace under `(`)**:

```csharp
items.Aggregate((acc, item) =>
                {
                    acc.Add(item);

                    return acc;
                });
```

**Async block lambda (brace under `async`)**:

```csharp
await Task.Run(async () =>
               {
                   await InitializeAsync();
                   await ProcessAsync();
               });
```

### 5.10 Anonymous Object Alignment

**Rule**: Multi-line anonymous objects align braces to the `new` keyword. Members are indented by **+1 level**.

```csharp
var person = new
             {
                 FirstName = "Max",
                 LastName = "Mustermann"
             };
```

## 6. Horizontal Spacing

### 6.1 Operator Spacing

**Rule**: Binary and assignment operators have **exactly one space** on each side.

```csharp
// Correct
var x = a + b;
x += 1;

// Incorrect
var x=a+b;
x +=1;
```

**Applies to**: All binary operators (`+`, `-`, `*`, `/`, `%`, `==`, `!=`, `<`, `>`, `<=`, `>=`, `&&`, `||`, `??`, `|`, `&`, `^`), all assignment operators (`=`, `+=`, `-=`, `*=`, etc.), and equals in variable declarations (`var x = 1`).

**Exception**: Multi-line expressions — spacing is not adjusted across line boundaries.

### 6.2 Comma Spacing

**Rule**: Commas are followed by **exactly one space**.

```csharp
// Correct
Method(a, b, c);

// Incorrect
Method(a,b,c);
Method(a , b , c);
```

**Exception**: Array rank specifiers (`int[,]`) — no space after comma.

### 6.3 Semicolons in For-Loops

**Rule**: Semicolons in `for` loop headers are followed by **exactly one space**.

```csharp
for (var i = 0; i < 10; i++)
```

### 6.4 Keyword Spacing

**Rule**: Control flow keywords are followed by **exactly one space** before the opening parenthesis or expression.

**Keywords**: `if`, `for`, `foreach`, `while`, `switch`, `catch`, `using`, `lock`, `return`, `throw`, `new`.

```csharp
if (condition)          // not: if(condition)
for (var i = 0; ...)    // not: for(var i = 0; ...)
return result;          // not: return(result) — unless intentional tuple
```

**Exceptions**:
- `return;` and `throw;` (before semicolon) — no space added.
- Target-typed `new()` and `new[]` — no space before `(` or `[`.

### 6.5 Parenthesis Spacing

**Rule**: No space after opening parenthesis or before closing parenthesis.

```csharp
// Correct
Method(a, b);

// Incorrect
Method( a, b );
```

### 6.6 Multiple Consecutive Spaces

**Rule**: Multiple consecutive spaces (within a line) are normalized to a **single space**.

## 7. Blank Lines

### 7.1 Blank Line Before Statements

**Rule**: The following statements must be preceded by a blank line — unless they are the **first statement in a block** (preceded by `{`, `:`, or file start):

| Statement | Exceptions |
|-----------|------------|
| `try` | — |
| `if` | Not when preceded by `else` (i.e., `else if` chains) |
| `while` | — |
| `do` | — |
| `using` | — |
| `foreach` | — |
| `for` | — |
| `return` | — |
| `goto` | — |
| `break` | Not inside `switch` sections (see §7.3) |
| `continue` | — |
| `throw` | — |
| `switch` | — |
| `checked` | — |
| `unchecked` | — |
| `fixed` | — |
| `lock` | — |
| `yield return`/`yield break` | Not when previous statement is also a `yield` statement |

### 7.2 Switch Case Braces

**Rule**: In `switch` statements, brace usage depends on whether **any** case section occupies more than one line (counting the statements, not the `case` label itself):

- If **any case** occupies more than one line (multiple statements, or a single multi-line statement), then **all cases** must use `{` and `}`.
- If **all cases** occupy exactly one line (a single single-line statement plus `break`), no `{` and `}` are used.

> **Note**: A multi-line method call (e.g., arguments wrapped across lines) counts as "more than one line" and triggers braces for all cases.

**All single-line cases — no braces**:

```csharp
switch (x)
{
    case 1:
        DoA();
        break;

    case 2:
        DoB();
        break;
}
```

**Any multi-line case — all cases get braces (braces +1 indent from `case`, `break` outside braces)**:

```csharp
switch (x)
{
    case 1:
        {
            DoA();
            DoB();
        }
        break;

    case 2:
        {
            DoC();
        }
        break;
}
```

### 7.3 Blank Line Before and After Break

**Rule**: A `break` statement inside a switch section has **no preceding blank line**. It always directly follows either:
- A closing brace `}` (when braces are used), or
- The single statement in a case section (when no braces are used)

**Rule**: A `break` statement must be **followed** by a blank line, unless it is the **last statement** in a block or switch section.

### 7.4 Maximum Consecutive Blank Lines

**Rule**: At most **one** blank line between any two code elements. Three or more consecutive blank lines are collapsed to one.

## 8. Region Formatting

### 8.1 Region Name Capitalization

**Rule**: `#region` descriptions must start with an **uppercase letter**.

```csharp
// Correct
#region Methods

// Incorrect
#region methods
```

### 8.2 Endregion Comment Synchronization

**Rule**: `#endregion` must include a comment matching the `#region` name.

```csharp
#region Methods

// ...

#endregion // Methods
```

### 8.3 Regions Without Names

**Rule**: Regions without a name are **skipped** (not modified).

## 9. Trailing Trivia & Cleanup

### 9.1 Trailing Whitespace

**Rule**: No trailing whitespace before end-of-line characters.

### 9.2 Blank Lines on Empty Lines

**Rule**: Blank lines must not contain whitespace characters (spaces/tabs are removed from blank lines).

### 9.3 End-of-File

**Rule**: Files must **not** end with a trailing newline character. The file ends with the last content character.

### 9.4 BOM Preservation

**Rule**: UTF-8 BOM markers (`\uFEFF`) at the file start are **preserved**.

### 9.5 No Blank Line After Opening Brace

**Rule**: There must be no blank line immediately after an opening brace `{`.

## 10. Comment Formatting

### 10.1 Comment Indentation

**Rule**: Comments must have the **same indentation** as the code they comment — specifically, the line immediately below the comment.

```csharp
public void Method()
{
    // This comment is about the if-statement below
    if (condition)
    {
        // This comment is about DoWork
        DoWork();
    }
}
```

**Applies to**: Single-line comments (`//`), multi-line comments (`/* */`), and XML documentation comments (`///`).

### 10.2 Blank Line Before Comments

**Rule**: A blank line must precede a comment — unless the comment is the **first element in a block** (preceded by `{`, `:`, or file start).

```csharp
public void Method()
{
    // No blank line needed — first in block
    DoSetup();

    // Blank line before this comment
    DoWork();

    // Blank line before this comment too
    DoCleanup();
}
```
