## Reihitsu.Core

### Session 1

- [Critical] Reihitsu.Core\ModifierOrderingUtilities.cs:145 — `GetRh7105Rank` ranks `partial` (7) before `readonly` (8), `volatile` (9), `async` (10), `extern` (11), `unsafe` (12) and `required` (13), but C# requires `partial` to be the last modifier, immediately before the type keyword or return type (CS0267). Valid, idiomatic declarations — `public readonly partial struct S`, `public unsafe partial class C`, `private async partial Task M()` — are therefore reported by `TryGetRh7105Violation` as violations, and `OrderModifiersForRh7105` demands the non-compiling order (`partial readonly`, `partial unsafe`, `partial async`), which the RH7105 code fix writes into the document. No test covers a `partial` combination with a higher-ranked modifier. Rank `partial` after all other modifiers (or pin its position during reordering). See open cross-check for the analyzer/fix confirmation.
- [Major] Reihitsu.Core\DeclarationModifierUtilities.cs:105 — `IsAccessibilityModifier` omits `SyntaxKind.FileKeyword` while the sibling helper `OrderingDeclarationUtilities.HasExplicitAccessibilityModifier` (OrderingDeclarationUtilities.cs:25) includes it: the same assembly now carries two contradictory definitions of "has accessibility". Consequence via RH2004/RH2005 (both call `HasAccessibilityModifier`): `file class C` is flagged although file-local types cannot legally take an accessibility modifier, and `AddAccessibilityModifier` would happily insert one (CS9052). Unify on one predicate and exempt `file`-modified declarations.
- [Major] Reihitsu.Core\CasingUtilities.cs:56 — type-parameter policy rejects digit-suffixed names: `IsTypeParameterName("T1")` is false because `IsPascalCase` demands an uppercase *letter* first, and `ToTypeParameterName("T1")` returns `"TT1"` because the prefix probe only accepts `char.IsUpper(input[1])` (a digit is not upper). `T1`/`T2` are the BCL-standard names for multiple type parameters (`Func<T1, T2>`), so RH4121 flags standard code and its fix renames `T1` → `TT1` (tests enshrine both: `CasingUtilitiesTests` rows `("T1", false)` and `("Tkey", "TTkey")`). Accept `T` + digits, and treat a digit after `T`/`t` as part of an existing prefix.
- [Major] Reihitsu.Core\OrderingDeclarationUtilities.cs:315 — `HasInitializer` misses `EventFieldDeclarationSyntax`: `public event EventHandler E = Handler;` carries a runtime initializer that executes in declaration order like a field initializer, but `ChangesInitializerExecutionOrder` returns false for it, so member-ordering fixes (RH7102/RH7103/RH7109/RH7110 family) may move an initialized event field past initialized fields — exactly the silent execution-order change this guard exists to prevent. Add the event-field case (variables with initializers, same static/instance context split).
- [Major] Reihitsu.Core (consolidation integrity) — the fix-wave consolidation recreated duplication *inside* Core: `FormattingSafetyUtilities.HasCommentsOrDirectives` (FormattingSafetyUtilities.cs:22) is byte-for-byte identical to `SyntaxNodeUtilities.HasCommentsOrDirectives` (SyntaxNodeUtilities.cs:21); `SyntaxNodeUtilities.IsComment` (SyntaxNodeUtilities.cs:41) duplicates `SyntaxTriviaUtilities.IsCommentTrivia` (SyntaxTriviaUtilities.cs:22); `RegionDirectiveUtilities` keeps a private `IsRegionDirective` (RegionDirectiveUtilities.cs:233) identical to the public `RegionDirectiveBlankLineUtilities.IsRegionDirective` (RegionDirectiveBlankLineUtilities.cs:21); and the single-line check exists three times (`SyntaxNodeUtilities.IsSingleLine`, `FormattingSafetyUtilities.IsSingleLineSpan`, `FormattingSafetyUtilities.AreAllSingleLine`). Callers can (and outside Core, do) pick different variants of the same policy. Collapse each family onto one helper.
- [Minor] Reihitsu.Core\SyntaxTriviaUtilities.cs:54 — `HasCommentDirectlyAbove` materializes the full document text via `token.SyntaxTree.GetText()` on every call and then applies string heuristics (`StartsWith("*")`, `EndsWith("*/")`) to decide comment adjacency instead of inspecting the leading-trivia structure; expensive in formatter/fix hot paths and the `*`-prefix heuristic is guesswork. Derive the answer from the trivia list.
- [Minor] Reihitsu.Core\UsingDirectiveOrderingUtilities.cs:131 — System-group detection is case-*insensitive* in `GetNamespaceGroupOrderKey` (`"SYSTEM"` maps into the System slot) but case-*sensitive* in `IsSystemNamespaceUsing` (UsingDirectiveOrderingUtilities.cs:46), so a root namespace differing only by case sorts into the System group while being classified `OtherNamespace` — RH7201 and the canonical order can disagree for such namespaces. Use one comparison (namespaces are case-sensitive; Ordinal on both is the safer choice).

## Reihitsu.Cli

### Session 1

- [Major] Reihitsu.Cli\Abstractions\DefaultFileSystem.cs:34 — the strict (throwing) decode added by #274 covers only UTF-8: BOM-detected UTF-16/UTF-32 files are decoded with the default replacement fallback (`Encoding.Unicode`, `BigEndianUnicode`, `UTF32Encoding`), so an invalid sequence (e.g. a lone surrogate inside a string literal) becomes U+FFFD silently, the file still parses, and a formatting write persists the corruption — the same silent-corruption class #274 fixed for UTF-8. Construct the UTF-16/32 encodings with throwing decoder fallbacks and route failures through the existing skip-with-warning path (the `DecoderFallbackException` handler in `FormatCommandHandler.ProcessFileAsync` already catches the base type).
- [Minor] Reihitsu.Cli\FormatCommandHandler.cs:200 — `IsInBuildOutputDirectory` matches only `/bin/`–`/obj/` with a *leading* separator on the raw enumerated path, and `CollectFiles` (FormatCommandHandler.cs:403) applies the filter before `GetFullPath`: `reihitsu-format bin` (relative) yields paths like `bin/x.cs` that escape the filter and get formatted, while the equivalent absolute invocation `/repo/bin` yields `/repo/bin/x.cs` and is skipped — same target, opposite outcome depending on path form. Also `OrdinalIgnoreCase` skips `/BIN/` on case-sensitive file systems. Normalize to full paths before filtering and decide the explicit-target semantics once.
- [Minor] Reihitsu.Cli\Diff\HunkBuilder.cs:30 — `Build`'s `originalLineCount`/`formattedLineCount` parameters are never used (dead parameters); the only caller (`DiffGenerator.Generate`) passes array lengths that the method ignores.
- [Minor] Reihitsu.Cli\Program.cs:19 — stale exit-code doc: the XML doc ("1 = formatting needed (--check)") and the `--check`-only usage text omit that `--dry-run` also exits 1 when changes are pending (`FormatCommandHandler.ExecuteAsync`, FormatCommandHandler.cs:187). Document the dry-run exit contract in both places.
